## 1. 技术选型总览

基于功能需求和非功能需求分析，OpenViking AI 运维系统的技术选型包括 **4 大类**：

1. **边缘计算框架**：容器化部署、资源管理
2. **LLM 集成框架**：LLM API 集成、Prompt 管理
3. **监控系统**：Prometheus + Grafana（已部署）
4. **部署工具**：Docker、Kubernetes Edge（可选）

## 2. 边缘计算框架选型

### 2.1 候选方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Docker** | 简单易用、生态成熟、轻量级 | 缺少编排、扩展性有限 |
| **Kubernetes Edge** | 自动编排、自愈能力、水平扩展 | 复杂、资源占用高 |
| **KubeEdge** | 边缘优化、离线支持 | 复杂、资源占用高 |
| **Docker Compose** | 简单易用、配置清晰 | 缺少编排、扩展性有限 |

### 2.2 选型分析

#### Docker Compose（推荐）

**优势**：
- ✅ 简单易用，学习成本低
- ✅ 配置清晰，易于管理
- ✅ 轻量级，资源占用低
- ✅ 已部署 OpenViking + VikingBot 使用 Docker
- ✅ 符合边缘计算设备限制（31Gi 内存）

**劣势**：
- ❌ 缺少自动编排能力
- ❌ 缺少自愈能力
- ❌ 扩展性有限

**适用场景**：
- 单设备或少设备（< 10）
- 简单的部署需求
- 低资源占用要求

#### Kubernetes Edge

**优势**：
- ✅ 自动编排能力
- ✅ 自愈能力
- ✅ 水平扩展
- ✅ 丰富的生态

**劣势**：
- ❌ 复杂，学习成本高
- ❌ 资源占用高（至少 2Gi）
- ❌ 不适合边缘计算设备限制

**适用场景**：
- 多设备（> 10）
- 复杂的部署需求
- 高可用性要求

#### KubeEdge

**优势**：
- ✅ 边缘优化
- ✅ 离线支持
- ✅ 轻量级

**劣势**：
- ❌ 复杂，学习成本高
- ❌ 资源占用仍然较高
- ❌ 社区相对小

**适用场景**：
- 多边缘设备
- 离线场景
- 低资源占用要求

### 2.3 选型决策

**推荐方案：Docker Compose**

**理由**：
1. 简单易用，学习成本低，快速上手
2. 轻量级，资源占用低，符合边缘计算限制
3. 已部署 OpenViking + VikingBot 使用 Docker，技术栈一致
4. 当前阶段（MVP）无需复杂的编排能力
5. 后续可以升级到 Kubernetes Edge（如有需要）

**配置示例**：
```yaml
version: '3.8'

services:
  # OpenViking Server
  openviking-server:
    image: vikingbot-cn-beijing.cr.volces.com/vikingbot/vikingbot:latest
    container_name: openviking-server
    entrypoint: ["/bin/sh","-lc"]
    command: ["pip uninstall -y chardet >/dev/null 2>&1 || true; exec /usr/local/bin/openviking serve --host 0.0.0.0 --port 1933 --config /root/.openviking/ov.conf"]
    network_mode: "service:vikingbot"
    volumes:
      - /home/scsun/.openviking:/root/.openviking
      - /home/scsun/.openviking-cli:/root/.openviking-cli
    restart: unless-stopped

  # VikingBot Gateway
  vikingbot:
    image: vikingbot-cn-beijing.cr.volces.com/vikingbot/vikingbot:latest
    container_name: vikingbot
    command: ["gateway"]
    ports:
      - "18791:18791"
    environment:
      - VIKING_STORAGE_VOLATILE=true
      - PYTHONWARNINGS=ignore:invalid escape sequence
    volumes:
      - /home/scsun/.openviking:/root/.openviking
      - /home/scsun/.openviking-cli:/root/.openviking-cli
      - /home/scsun/.vikingbot:/root/.vikingbot
      - /home/scsun/.vikingbot-cli:/root/.vikingbot-cli
    restart: unless-stopped

  # AI 运维服务
  aiops-service:
    build: ./src/aiops-service
    container_name: aiops-service
    ports:
      - "8080:8080"
    environment:
      - OPENVIKING_SERVER_URL=http://openviking-server:1933
      - VOLCENGINE_API_KEY=${VOLCENGINE_API_KEY}
    depends_on:
      - openviking-server
    restart: unless-stopped
```

## 3. LLM 集成框架选型

### 3.1 候选方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| **直接调用 API** | 简单直接、易于实现 | 缺少抽象、难以扩展 |
| **OpenAI SDK** | 标准化、生态成熟 | 仅支持 OpenAI 兼容 |
| **LangChain** | 丰富功能、易于扩展 | 复杂、资源占用高 |
| **VikingBot 集成** | 已集成、开箱即用 | 仅限 VikingBot 内 |

### 3.2 选型分析

#### VikingBot 集成（推荐）

**优势**：
- ✅ 已集成火山引擎豆包 LLM
- ✅ 开箱即用，无需额外配置
- ✅ OpenViking 集成，知识管理方便
- ✅ 轻量级，资源占用低
- ✅ 已部署，无需额外部署

**劣势**：
- ❌ 仅限 VikingBot 内使用
- ❌ 不支持外部应用调用

**适用场景**：
- 使用 VikingBot 进行智能运维
- 利用 OpenViking 进行知识管理
- 快速开发智能运维功能

#### 直接调用 API

**优势**：
- ✅ 简单直接，易于实现
- ✅ 灵活性高，可以自定义逻辑
- ✅ 可以在任何应用中调用

**劣势**：
- ❌ 缺少抽象，需要处理底层细节
- ❌ 需要自己管理 API Key、限流等

**适用场景**：
- 外部应用需要调用 LLM
- 需要自定义逻辑
- 需要灵活性

#### LangChain

**优势**：
- ✅ 丰富功能（Prompt 管理、Chain、Agent）
- ✅ 易于扩展，支持多种 LLM
- ✅ 生态成熟，社区活跃

**劣势**：
- ❌ 复杂，学习成本高
- ❌ 资源占用高（需要 Python 环境）
- ❌ 不符合边缘计算限制

**适用场景**：
- 复杂的 LLM 应用
- 需要多 LLM 集成
- 需要丰富的 LLM 功能

### 3.3 选型决策

**推荐方案：VikingBot 集成**

**理由**：
1. 已集成火山引擎豆包 LLM，开箱即用
2. OpenViking 集成，知识管理方便
3. 轻量级，资源占用低，符合边缘计算限制
4. 快速开发，无需额外配置

**架构示例**：
```python
# AI 运维服务调用 VikingBot LLM
import requests

class AIOpsLLMClient:
    def __init__(self, vikingbot_url="http://vikingbot:18791"):
        self.vikingbot_url = vikingbot_url

    async def analyze_logs(self, logs):
        """分析日志"""
        prompt = f"""
        请分析以下日志，识别错误和异常：
        
        {logs}
        
        请输出：
        1. 错误和异常列表
        2. 错误和异常的严重级别
        3. 错误和异常的可能原因
        4. 错误和异常的修复建议
        """
        response = await self._call_llm(prompt)
        return response

    async def detect_anomaly(self, metrics):
        """检测异常"""
        prompt = f"""
        请分析以下监控指标，检测异常：
        
        {metrics}
        
        请输出：
        1. 异常检测结果
        2. 异常类型和严重级别
        3. 异常的可能原因
        4. 异常的处理建议
        """
        response = await self._call_llm(prompt)
        return response

    async def diagnose_fault(self, fault_info):
        """诊断故障"""
        prompt = f"""
        请诊断以下故障，提供根因分析和修复建议：
        
        {fault_info}
        
        请输出：
        1. 故障根因分析
        2. 故障修复建议
        3. 故障预防措施
        """
        response = await self._call_llm(prompt)
        return response

    async def _call_llm(self, prompt):
        """调用 VikingBot LLM"""
        response = requests.post(
            f"{self.vikingbot_url}/v1/chat/completions",
            json={
                "model": "openai/doubao-seed-2-0-pro-260215",
                "messages": [
                    {"role": "system", "content": "你是一个专业的运维专家，擅长日志分析、异常检测和故障诊断。"},
                    {"role": "user", "content": prompt}
                ],
                "temperature": 0.7,
                "max_tokens": 2048
            }
        )
        return response.json()["choices"][0]["message"]["content"]
```

## 4. 监控系统选型

### 4.1 候选方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Prometheus + Grafana** | 已部署、生态成熟、功能强大 | 学习成本高 |
| **ELK Stack** | 日志分析强大 | 资源占用高 |
| **Zabbix** | 传统运维监控 | 功能单一、界面老旧 |

### 4.2 选型分析

#### Prometheus + Grafana（推荐）

**优势**：
- ✅ 已部署，无需额外部署
- ✅ 生态成熟，功能强大
- ✅ 支持多种数据源
- ✅ 丰富的 Grafana 面板
- ✅ 强大的告警规则引擎

**劣势**：
- ❌ 学习成本高
- ❌ 需要维护

**适用场景**：
- 监控数据采集和存储
- 数据可视化
- 告警管理

#### ELK Stack

**优势**：
- ✅ 日志分析强大
- ✅ 支持全文搜索
- ✅ 丰富的查询语法

**劣势**：
- ❌ 资源占用高（至少 8Gi）
- ❌ 不符合边缘计算限制

**适用场景**：
- 日志收集和分析
- 不推荐用于边缘计算设备

#### Zabbix

**优势**：
- ✅ 传统运维监控
- ✅ 功能稳定

**劣势**：
- ❌ 功能单一
- ❌ 界面老旧
- ❌ 不支持现代监控场景

**适用场景**：
- 传统运维监控
- 不推荐用于现代智能运维

### 4.3 选型决策

**推荐方案：Prometheus + Grafana**

**理由**：
1. 已部署，无需额外部署
2. 生态成熟，功能强大
3. 符合现代监控需求

**集成示例**：
