## 1. 架构决策记录（ADR）

### 1.1 双层架构决策（ADR-001）

**状态**：已接受

**背景**：
项目采用双层架构：研发层（OpenClaw + Claude Code）和运维层（OpenViking + VikingBot + AI 服务）。

**决策**：
采用双层架构，研发层和运维层职责分离。

**理由**：
1. **职责清晰**：研发层负责开发和测试，运维层负责生产运行
2. **数据分离**：研发知识和运维知识严格分离
3. **风险控制**：研发层不直接访问生产环境，降低风险
4. **可维护性**：研发层和运维层可以独立演进

**后果**：
- **正面**：职责清晰、风险可控、可维护性好
- **负面**：增加开发和运维沟通成本

---

### 1.2 模块划分决策（ADR-002）

**状态**：已接受

**背景**：
运维层需要支持多个运维能力，包括边缘计算、网络运维、服务器运维、AI 智能运维、监控告警、运维知识管理。

**决策**：
采用模块化架构，将运维层划分为 6 大模块：
1. 边缘计算模块
2. 网络运维模块
3. 服务器运维模块
4. AI 智能运维模块
5. 监控告警模块
6. 运维知识管理模块

**理由**：
1. **职责单一**：每个模块负责一类运维能力
2. **低耦合**：模块间通过接口交互，降低耦合度
3. **高内聚**：每个模块内部功能高度相关
4. **可扩展**：可以独立扩展每个模块

**后果**：
- **正面**：职责清晰、低耦合、高内聚、可扩展
- **负面**：模块多、交互复杂

---

### 1.3 技术选型决策（ADR-003）

**状态**：已接受

**背景**：
需要选择适合边缘计算设备的技术栈，包括容器化、编排、LLM 集成、监控、告警。

**决策**：
采用以下技术栈：
1. **容器化**：Docker
2. **编排**：Docker Compose
3. **LLM 集成**：VikingBot 集成（无需开发）
4. **监控**：Prometheus + Grafana
5. **告警**：AlertManager
6. **日志**：Loki

**理由**：
1. **简单易用**：Docker 和 Docker Compose 简单易用，学习成本低
2. **轻量级**：符合边缘计算设备限制（31Gi 内存）
3. **已部署**：Prometheus、Grafana、Loki、AlertManager 已部署，无需额外部署
4. **开箱即用**：VikingBot 已集成 LLM，开箱即用
5. **生态成熟**：所选技术栈生态成熟，有丰富的文档和社区支持

**后果**：
- **正面**：简单易用、轻量级、已部署、开箱即用、生态成熟
- **负面**：缺少自动编排能力、扩展性有限

---

### 1.4 接口设计决策（ADR-004）

**状态**：已接受

**背景**：
模块间需要定义清晰的接口，包括模块间接口和外部接口。

**决策**：
采用 RESTful API 作为接口协议，使用 JSON 作为数据格式。

**理由**：
1. **简单易用**：RESTful API 简单易用，学习成本低
2. **生态成熟**：RESTful API 生态成熟，有丰富的工具和框架
3. **跨平台**：RESTful API 跨平台，易于集成
4. **可扩展**：RESTful API 易于扩展，支持多种数据格式

**后果**：
- **正面**：简单易用、生态成熟、跨平台、可扩展
- **负面**：实时性较差、不支持长连接

---

### 1.5 数据模型设计决策（ADR-005）

**状态**：已接受

**背景**：
需要设计清晰的数据模型，包括监控数据、日志数据、告警数据、运维知识数据。

**决策**：
采用关系型数据库作为主数据存储，使用 PostgreSQL。

**理由**：
1. **成熟稳定**：PostgreSQL 成熟稳定，功能强大
2. **支持复杂查询**：PostgreSQL 支持复杂查询、事务、索引
3. **数据一致性**：PostgreSQL 保证数据一致性
4. **可扩展性**：PostgreSQL 支持水平扩展

**后果**：
- **正面**：成熟稳定、支持复杂查询、数据一致性、可扩展性
- **负面**：需要部署和维护 PostgreSQL

---

### 1.6 部署架构决策（ADR-006）

**状态**：已接受

**背景**：
需要在边缘计算设备上部署所有服务，包括 OpenViking Server、VikingBot、AI 服务、监控、告警。

**决策**：
采用 Docker Compose 作为部署工具，所有服务容器化部署。

**理由**：
1. **简单易用**：Docker Compose 简单易用，学习成本低
2. **轻量级**：符合边缘计算设备限制（31Gi 内存）
3. **已部署**：OpenViking Server 和 VikingBot 已使用 Docker 部署，技术栈一致
4. **快速部署**：Docker Compose 支持快速部署，部署时间 < 10 分钟

**后果**：
- **正面**：简单易用、轻量级、已部署、快速部署
- **负面**：缺少自动编排能力、扩展性有限
