# 01-项目总览

## 项目名称
OpenViking AI 运维系统

## 背景
当前已部署 OpenViking 项目（上下文数据库）和 OpenViking Bot（AI 智能体），运行在边缘计算设备上（172.16.100.101:22）。

研发侧采用 **AIDevault 开发框架**，通过 OpenClaw + Claude Code 实现标准化、可追溯的研发流程。

OpenViking 环境用于**运维能力建设**，不纳入研发主流程。运维产品在运行时，会通过 OpenViking Bot 与 LLM 进行智能运维推理。

## 项目目标

### 研发目标
基于 AIDevault 框架，开发一个基于边缘计算设备的 AI 运维系统，实现：
- 边缘计算模块：AI 模型部署到边缘设备
- 网络运维模块：网络状态监控、设备健康度监控
- AI 模型模块：时间序列预测、异常检测、分类模型
- 监控告警模块：设备健康度、网络性能、AI 模型性能

### 运维目标
在研发交付的运维系统基础上，实现：
- 实时监控网络状态和设备健康度
- AI 驱动的故障预测和自动诊断
- 智能告警和自动化运维决策
- 运维经验的自动沉淀与学习

## 非目标
- OpenViking / OpenViking Bot 不参与研发主流程
- 不直接接入生产运维环境进行开发调试
- 不在研发阶段实现全自动运维决策
- 不在第一阶段做多设备集群编排
- 不做"让运维 Bot 自由执行"的无护栏模式

## 双层架构

### 研发层（开发与管理）
```
OpenClaw（总控与任务编排）
    ↓
Claude Code（代码修改与执行）
    ↓
Obsidian（项目知识沉淀）
    ↓
Git / CI（质量门禁）
    ↓
部署交付
```

### 运维层（生产环境）
```
OpenViking（上下文管理 / AGFS）
    ↓
OpenViking Bot（LLM 智能运维推理）
    ↓
边缘设备群（网络监控 / 自动运维）
```

## 核心原则

### 研发层原则
1. **OpenClaw 负责总控与任务编排** - 任务下发、进程管理、会话协调
2. **Claude Code 负责代码修改与执行** - 代码开发、测试、调试
3. **Obsidian 负责项目知识沉淀** - 需求分析、架构设计、最佳实践记录
4. **Git / CI 负责质量门禁** - 代码审查、自动化测试、质量门禁

### 运维层原则
1. **OpenViking 不参与研发主流程** - 仅作为运维环境的上下文数据库
2. **OpenViking Bot 负责 LLM 智能运维推理** - 运维决策、故障诊断、自动修复建议
3. **运维产品在运行时与 LLM 交互** - 通过 Bot 接口进行智能运维推理
4. **运维经验沉淀在 OpenViking** - 不回流到研发层，只在运维环境内循环

### 边界原则
- OpenClaw / Claude Code 不直接访问生产运维环境（除非部署）
- OpenViking / OpenViking Bot 不参与代码开发和研发流程
- 研发与运维的数据严格分离（研发知识 vs 运维知识）

## 成功标准

### 研发成功标准
1. OpenClaw 能稳定调度 Claude Code 完成 OpenViking AI 运维系统开发
2. Claude Code 有明确边界、限制和测试要求
3. 任务过程不依赖聊天记忆，关键结论均落地 Obsidian
4. 产出能沉淀为任务记录、ADR、复盘和发布说明
5. 后续新运维模块能够快速复用研发框架

### 运维成功标准
1. OpenViking 能有效管理运维知识和上下文
2. OpenViking Bot 能通过 LLM 进行准确的运维推理
3. 实现实时监控和智能告警
4. 运维经验能自动沉淀和复用
5. 系统可用性 > 99.9%

## 关键风险

### 研发风险
1. Claude Code 操作边界不清晰，导致高风险文件被误改
2. 项目知识只存在聊天记录中，缺乏长期可追溯性
3. 测试与质量门禁不完整，产出不稳定
4. 边缘计算环境的资源限制导致开发困难
5. 研发与运维的边界不清晰导致职责混乱

### 运维风险
1. OpenViking 知识库不完整，导致运维推理不准确
2. 智能运维决策不可靠，导致误操作
3. 实时性要求高（< 100ms），导致推理延迟问题
4. 网络不稳定导致运维决策失效
5. 安全性要求高（数据不离开边缘设备），导致隐私泄露

### 应对策略
- **研发侧**：用 OpenClaw 固化任务入口和会话节奏；用 Claude Code hooks 做强约束；用 Obsidian 做单一项目知识中台；用 Git / CI 做最终门禁
- **运维侧**：用 OpenViking 存储所有运维知识和经验；用 Bot 接口进行 LLM 推理；用边缘设备进行实时监控和决策；严格控制数据隐私和访问权限

## 关键约束

### 边缘计算限制
- 设备能力有限（CPU、内存、存储限制）
- 网络不稳定，带宽限制
- 能耗优化，电池寿命考虑

### 实时性要求
- AI 决策延迟 < 100ms
- 处理吞吐量 > 100 请求/秒
- 系统可用性 > 99.9%

### 安全性要求
- 数据隐私：数据不离开边缘设备
- 加密传输：所有网络通信加密
- 访问控制：严格的访问控制（RBAC）

### 可扩展性要求
- 水平扩展：支持添加更多边缘设备
- 模型更新：云端训练、边缘部署
- 功能扩展：模块化设计，易于扩展
