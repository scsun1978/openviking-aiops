# OpenClaw 任务下发模板

> 此模板用于 OpenClaw 向 Claude Code 下发任务，确保信息完整、目标清晰

---

## 标准任务格式

### 基本信息

```yaml
项目：[项目名称]
任务编号：DEV-XXX
任务名称：[简短任务名称]
优先级：[高/中/低]
目标仓库：[仓库路径或 URL]
目标分支：[main / develop / feature-xxx]
```

### 任务目标

> 用 1-2 句话清晰描述任务的核心目标

[目标描述]

### 范围

> 列出任务包含的工作内容

- [ ] 范围项 1
- [ ] 范围项 2
- [ ] 范围项 3

### 非目标

> 明确列出任务不包含的内容

- 不做 xxx
- 不修改 xxx

### 验收标准

> 明确的可测试的完成条件

1. [ ] 可测试条件 1
2. [ ] 可测试条件 2
3. [ ] 可测试条件 3

### 风险边界

> 明确禁止的操作和高风险区域

- 禁止直接修改 main/master 分支
- 禁止修改生产数据库
- 禁止删除 xxx

## 执行指令

### 标准执行流程

请按以下步骤执行：

1. **只读分析**（必须）
   - 读取相关代码文件
   - 分析现有架构和依赖
   - 列出涉及模块和文件
   - 识别潜在风险点

2. **方案设计**（必须）
   - 给出实施方案
   - 列出需要修改的文件
   - 说明技术选型原因
   - 列出测试策略

3. **实施确认**（必须）
   - 等待方案确认
   - 未经确认不得开始编码

4. **编码实现**
   - 按方案实施
   - 遵循代码规范
   - 添加必要注释
   - 更新相关文档

5. **验证测试**
   - 执行 lint
   - 执行单元测试
   - 执行集成测试
   - 记录测试结果

6. **输出总结**（必须）
   - 修改文件清单
   - 主要变更摘要
   - 测试执行结果
   - 遗留问题清单
   - PR 摘要文案
   - 回归建议
   - Obsidian 任务纪要

### 安全要求

- **只读优先**：先分析再动手
- **分支保护**：绝不直接改 main/master
- **风险确认**：高风险操作需确认
- **可回滚性**：所有修改应可回滚
- **测试覆盖**：修改必须有测试

### 输出要求

#### 分析阶段输出

```yaml
涉及模块：
  - 模块 A
  - 模块 B

涉及文件：
  - path/to/file1.js
  - path/to/file2.js

风险点：
  - 风险 1：[描述和应对]
  - 风险 2：[描述和应对]

实施方案：
  1. 步骤 1
  2. 步骤 2
  3. 步骤 3
```

#### 实施阶段输出

```yaml
已修改文件：
  新增：
    - path/to/new/file.js
  修改：
    - path/to/modified/file.js
  删除：
    - path/to/deleted/file.js

主要变更：
  1. 变更 1
  2. 变更 2

测试命令：
  - npm test
  - npm run lint

测试结果：
  - ✅ 通过：X 个测试
  - ❌ 失败：Y 个测试
  - ⏭ 跳过：Z 个测试

遗留问题：
  - 问题 1：[描述]
  - 问题 2：[描述]
```

#### 收口阶段输出

```yaml
PR 摘要：
  标题：[简短标题]
  描述：[变更描述]
  测试：[测试说明]

回归建议：
  - 测试点 1：[具体内容]
  - 测试点 2：[具体内容]
  - 回归范围：[受影响模块]

部署建议：
  - 飨险级别：[L1/L2/L3]
  - 部署方式：[直接/分阶段]
  - 回滚方案：[具体步骤]

需写入文档的结论：
  - [文档项 1]
  - [文档项 2]
```

## 集成说明

### 与任务文件关联

1. OpenClaw 下发任务时，先在 `tasks/DEV-XXX.md` 创建任务文件
2. 使用 `templates/任务模板.md` 作为基础
3. 将此模板内容追加到任务文件中

### 与 Claude Code 关联

1. Claude Code 读取任务文件
2. 按照执行流程逐步执行
3. 将执行记录回写到任务文件的"执行记录"部分

### 与 ADR 关联

如果任务过程中有重要技术决策：
1. 在 `07-决策记录-ADR.md` 中创建 ADR
2. 在任务文件的"决策记录"部分添加引用
3. 在输出中说明需要记录的决策

## 快速下发示例

### 简单任务

```
项目：my-app
任务编号：DEV-005
目标：添加用户登录功能
范围：登录表单、API 调用、状态管理
验收标准：
1. 用户可以输入用户名密码
2. 点击登录后调用 API
3. 登录成功后跳转到首页
4. 登录失败显示错误提示
请按标准流程执行。
```

### 复杂任务

直接复制 `templates/任务模板.md` 填写完整内容后下发。

---

## 注意事项

1. **目标清晰**：验收标准必须可测试
2. **范围明确**：非目标避免范围蔓延
3. **风险前置**：高风险操作提前说明
4. **输出完整**：确保所有输出项都完成
