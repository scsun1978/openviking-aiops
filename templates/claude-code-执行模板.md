# Claude Code 执行模板

> 此模板定义 Claude Code 执行任务时的标准流程和输出格式

---

## 执行原则

### 核心原则

1. **只读优先**：先分析理解，再动手修改
2. **风险意识**：识别风险，提前预案
3. **分支保护**：绝不直接修改 main/master
4. **可回滚性**：所有修改应可回滚
5. **测试覆盖**：修改必须有测试验证
6. **文档同步**：代码变更同步更新文档

### 安全要求

- **只读分析**：使用 Read、Grep、Glob 等只读工具先理解代码
- **禁止操作**：
  - ❌ 直接修改 main/master 分支
  - ❌ 删除数据库或数据文件
  - ❌ 修改生产配置
  - ❌ 推送未经测试的代码
- **高风险操作**：以下操作需确认
  - ⚠️ 删除文件/目录
  - ⚠️ 修改核心依赖
  - ⚠️ 数据库迁移
  - ⚠️ 部署到生产环境

## 执行流程

### 阶段 1：只读分析（必须）

**目标**：理解现有代码，识别风险点

**执行步骤**：

1. **读取任务文件**
   ```bash
   # 读取 tasks/DEV-XXX.md
   理解任务目标、范围、验收标准
   ```

2. **分析代码结构**
   - 使用 Glob 查找相关文件
   - 使用 Read 读取关键文件
   - 使用 Grep 搜索相关代码

3. **识别风险点**
   - 高风险目录：`/config/`, `/database/`, `/scripts/`
   - 高风险文件：主配置文件、数据库文件、部署脚本
   - 高风险操作：删除、覆盖、批量修改

4. **输出分析结果**
   ```yaml
   # 阶段 1 分析结果

   任务理解：
     任务编号：DEV-XXX
     任务目标：[目标描述]
     验收标准：[标准列表]

   涉及模块：
     - 模块 A：[说明]
     - 模块 B：[说明]

   涉及文件：
     - path/to/file1.js：[说明]
     - path/to/file2.js：[说明]

   风险评估：
     - L1（低）：[风险项]
     - L2（中）：[风险项]
     - L3（高）：[风险项]

   技术方案：
     1. [步骤 1]
     2. [步骤 2]
     3. [步骤 3]

   测试策略：
     - 单元测试：[说明]
     - 集成测试：[说明]
   ```

### 阶段 2：方案确认（必须）

**目标**：确保方案正确，获得确认

**执行步骤**：

1. **展示方案**
   - 说明技术选型
   - 列出修改文件清单
   - 说明测试计划

2. **等待确认**
   - 未经确认不开始编码
   - 可根据反馈调整方案

3. **输出确认结果**
   ```yaml
   # 阶段 2 方案确认

   实施方案：
     - 技术栈：[框架/库]
     - 架构模式：[设计模式]
     - 关键决策：[决策说明]

   修改计划：
     新增文件：
       - path/to/new/file.js：[说明]
     修改文件：
       - path/to/file.js：[修改说明]
     删除文件：
       - path/to/file.js：[删除原因]

   风险预案：
     - 风险 1：[应对方案]
     - 风险 2：[应对方案]

   状态：
     - ✅ 方案已确认，开始实施
     - ⏳ 等待确认中
   ```

### 阶段 3：编码实现（确认后）

**目标**：按方案实施，保证质量

**执行步骤**：

1. **创建/修改文件**
   - 使用 Write 创建新文件
   - 使用 Edit 修改现有文件
   - 遵循代码规范

2. **代码审查**
   - 自查代码质量
   - 检查安全漏洞
   - 验证逻辑正确性

3. **输出实施结果**
   ```yaml
   # 阶段 3 实施结果

   已完成修改：
     新增文件：
       ✅ path/to/new/file.js
     修改文件：
       ✅ path/to/file.js（修改内容说明）

   主要变更：
     1. [变更 1]
     2. [变更 2]
     3. [变更 3]

   代码检查：
     - ✅ 代码规范检查通过
     - ✅ 安全检查通过
     - ✅ 逻辑验证通过
   ```

### 阶段 4：测试验证（必须）

**目标**：确保代码质量，无回归问题

**执行步骤**：

1. **执行测试**
   ```bash
   # 示例测试命令
   npm test
   npm run lint
   npm run build
   ```

2. **记录测试结果**
   ```yaml
   # 阶段 4 测试结果

   测试执行：
     命令 1：npm test
       结果：✅ 通过
       输出：
         - 测试套件：1 passed, 1 total
         - 测试用例：6 passed, 6 total

     命令 2：npm run lint
       结果：✅ 通过
       输出：No linting errors

     命令 3：npm run build
       结果：✅ 通过
       输出：Build successful

   测试覆盖：
     - 语句覆盖：XX%
     - 分支覆盖：XX%
     - 函数覆盖：XX%

   验收标准验证：
     - ✅ 标准 1：[验证结果]
     - ✅ 标准 2：[验证结果]
     - ✅ 标准 3：[验证结果]
   ```

### 阶段 4.5：发布前检查（必须）

**目标**：按照发布检查清单进行质量验证，确保可安全发布

**执行步骤**：

1. **执行 L1 基础检查清单**
   - 按照 `docs/11-发布检查清单.md` 进行逐项检查
   - 记录每项检查的结果

2. **输出检查结果**
   ```yaml
   # 阶段 4.5 发布前检查结果

   代码质量检查：
     Lint 检查：
       - ✅ 通过（命令：npm run lint）
       - 输出：无 linting errors
     代码格式检查：
       - ✅ 通过（命令：npm run format）
       - 输出：格式化完成
     代码审查：
       - ✅ 通过（已自审）
       - 检查项：无安全漏洞、无调试代码、无注释代码

   测试验证：
     单元测试：
       - ✅ 通过（命令：npm test）
       - 测试套件：X passed, X total
       - 测试覆盖率：XX%
     关键功能测试：
       - ✅ 功能 A：[测试结果]
       - ✅ 功能 B：[测试结果]
     兼容性测试：
       - ⏭️ 不适用 / ✅ 通过

   构建验证：
     构建检查：
       - ✅ 通过（命令：npm run build）
       - 输出：构建成功
     依赖检查：
       - ✅ 通过
       - 无新增未审核依赖

   文档完整性：
     README 更新：
       - ✅ 已更新 / ⏭️ 无需更新
     CHANGELOG 更新：
       - ✅ 已更新
     API 文档：
       - ⏭️ 不适用 / ✅ 已更新
     代码注释：
       - ✅ 关键代码有注释

   变更评估：
     影响范围：
       - 受影响模块：[模块列表]
       - 受影响依赖方：[依赖方列表]
     风险评估：
       - L1（低）：[风险项]
       - L2（中）：[风险项]
       - L3（高）：[风险项]
     回归计划：
       - 回归测试点：[测试点列表]
       - 回归范围：[范围说明]

   检查结论：
     - ✅ 所有 L1 检查项通过，可以发布
     - / ⚠️ 存在问题，需修复后重新检查
       - 问题描述：[具体问题]
   ```

3. **处理检查结果**
   - 如所有检查项通过，继续到阶段 5
   - 如有检查项失败，先修复问题，再重新执行检查

### 阶段 5：输出总结（必须）

**目标**：生成完整输出，支持文档回写和变更评估

**执行步骤**：

1. **生成文件清单**
   ```yaml
   修改文件清单：
     新增文件：
       - path/to/new/file.js
     修改文件：
       - path/to/modified/file.js
     删除文件：
       - path/to/deleted/file.js

     总计：X 个新增，Y 个修改，Z 个删除
   ```

2. **生成 PR 摘要**
   ```yaml
   PR 摘要：
     标题：DEV-XXX: [简短标题]
     描述： |
       ## 变更内容
       - [变更 1]
       - [变更 2]

       ## 测试
       - [测试说明]

       ## 文档
       - [文档更新]

       ## 变更评估
       - 影响范围：[模块/功能列表]
       - 风险等级：[低/中/高]
   ```

3. **生成回归建议**
   ```yaml
   回归建议：
     风险评估：
       - L1：无
       - L2：[影响范围]
       - L3：[高风险操作]

     回归测试点：
       1. [测试点 1]
       2. [测试点 2]

     回归范围：[受影响模块]

     部署建议：
       - 部署方式：[直接/分阶段]
       - 风险等级：[低/中/高]
       - 回滚方案：[具体步骤]

     发布后监控：
       - 监控指标：[关键指标]
       - 告警设置：[告警条件]
       - 验证时间：[发布后 X 小时]
   ```

4. **生成任务纪要**
   ```markdown
   ## DEV-XXX [任务名称] 执行纪要

   ### 基本信息
   - 开始时间：[时间]
   - 完成时间：[时间]
   - 执行状态：[DONE/BLOCKED]

   ### 主要变更
   [变更摘要]

   ### 发布检查结果
   [检查清单执行结果]

   ### 测试结果
   [测试结果]

   ### 遗留问题
   - [问题 1]
   - [问题 2]

   ### 决策记录
   如有重要决策，记录 ADR 编号

   ### 后续建议
   - [建议 1]
   - [建议 2]
   ```

## 输出检查清单

在任务完成前，确保以下所有项都已输出：

- [ ] 分析阶段：模块、文件、风险、方案
- [ ] 实施阶段：修改清单、主要变更
- [ ] 测试阶段：测试命令、测试结果
- [ ] 遗留问题：未解决问题清单
- [ ] PR 摘要：标题、描述
- [ ] 回归建议：风险评估、测试点
- [ ] 任务纪要：可写入 Obsidian 的格式
- [ ] Git 提交：建议的 commit message
