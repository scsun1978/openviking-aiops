## 核心原则

### 研发层原则
1. **OpenClaw 负责总控与任务编排** - 任务下发、进程管理、会话协调
2. **Claude Code 负责代码修改与执行** - 代码开发、测试、调试
3. **Obsidian 负责项目知识沉淀** - 需求分析、架构设计、最佳实践记录
4. **Git / CI 负责质量门禁** - 代码审查、自动化测试、质量门禁

### 运维层原则
1. **OpenViking 不参与研发主流程** - 仅作为运维环境的上下文数据库
2. **OpenViking Bot 负责 LLM 智能运维推理** - 运维决策、故障诊断、自动修复建议
3. **运维产品在运行时与 LLM 交互** - 通过 Bot 接口进行智能运维推理
4. **运维经验沉淀在 OpenViking** - 不回流到研发层，只在运维环境内循环

**核心原则**：

- 轻量级，资源占用低，符合边缘计算限制

## 非功能需求

### 边缘计算限制
- CPU：> 20 核心，使用率 < 70%
- 内存：> 20Gi，使用率 < 80%
- 存储：> 100Gi，使用率 < 90%
- 网络：带宽 > 150Mbps，延迟 < 100ms
- 能耗：< 5kWh/天

### 实时性要求
- AI 智能运维决策延迟：< 100ms
- 处理吞吐量：> 100 请求/秒
- 系统可用性：> 99.9%
- 告警响应时间：< 1 分钟

### 安全性要求
- 数据隐私：数据不离开边缘设备
- 加密传输：所有网络通信加密
- 访问控制：严格的 RBAC

### 可扩展性要求
- 水平扩展：支持多设备
- LLM 模型更新：通过 LLM API（无需本地训练）
- 功能扩展：模块化设计，易于扩展

## 注意事项

### 研发层注意事项
1. OpenClaw / Claude Code 不直接访问生产运维环境
2. OpenViking / OpenViking Bot 不参与研发流程
3. 研发知识和运维知识严格分离

### 运维层注意事项
1. 所有服务容器化部署（Docker）
2. 使用 Docker Compose 管理服务
3. 使用现有 LLM（火山引擎豆包）进行智能运维推理
4. 不开发 AI 模型训练能力

### 开发注意事项
1. 代码风格：遵循 PEP 8（Python）或 ESLint（TypeScript）
2. 提交信息：清晰、简洁、规范
3. 文档更新：代码和文档同步更新
4. 测试覆盖：单元测试覆盖率 > 80%
